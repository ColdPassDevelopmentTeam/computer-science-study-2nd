# 2안 4주 고강도 기술 인터뷰 커리큘럼

[2안 운영 규칙](../README.md)

## 커리큘럼 개요

매주 기술 이름이 아니라 판단해야 할 대표 질문 하나에서 시작합니다. 첫 답변은 결론부터
말하고, 꼬리질문에서 동작·조건·대안·실패·운영 책임으로 깊이를 늘립니다.

4주는 완료한 4개 세션을 뜻합니다. 세션이 취소되면 주차를 건너뛰지 않고 다음 일정에 같은
주차를 진행합니다.

## 시작 기준

참여자가 그 주의 공개 키워드를 자료 없이 2~3분 설명할 수 있어야 합니다. 처음 접하는
키워드가 대부분이거나 내부 동작을 설명하기 어렵다면 [1안](../../plan-1/)을 먼저 진행합니다.
1안과 2안을 같은 세션에서 섞지 않습니다.

## 4주 질문

| 주차 | 이번 주 질문 | 도착해야 할 판단 |
|---:|---|---|
| [1주차](./week-01.md) | Virtual Thread로 바꾸면 동시성 문제와 성능 병목이 해결되는가? | 실행 모델, 공유 상태와 전환 비용을 나누어 판단 |
| [2주차](./week-02.md) | Timeout 뒤 재시도해도 중복 처리와 DB 경합을 막을 수 있는가? | 원격 호출 결과와 데이터 일관성을 함께 통제 |
| [3주차](./week-03.md) | `@Transactional`을 붙였는데도 데이터가 예상과 다르게 저장되는 이유는 무엇인가? | Proxy·persistence context·flush·commit 경계를 구분 |
| [4주차](./week-04.md) | MSA로 나누면 배포·장애·보안 문제가 실제로 줄어드는가? | 분리 이익과 분산 비용, 운영 책임을 조건에 맞게 비교 |

## 주차 문서 읽는 순서

1. 제목의 대표 질문에 결론부터 답합니다.
2. 공개된 시스템 전제와 필수 키워드 안에서 답변을 준비합니다.
3. `답변을 확장할 방향`을 외울 목차가 아닌 꼬리질문 범위로 사용합니다.
4. 질문자는 규모, 부하, 장애 조건 또는 제약을 하나 이상 바꿔 실제 질문을 만듭니다.
5. 실제 질문과 예상 답변은 면접 대상자에게 미리 공개하지 않습니다.

## Depth 기준

| Depth | 답변에서 확인할 내용 |
|---:|---|
| 1 | 질문에 대한 결론 |
| 2 | 내부 동작 순서 |
| 3 | 결론이 성립하는 조건과 경계 |
| 4 | 주어진 시스템에 적용 |
| 5 | 대안과 trade-off |
| 6 | 실패 조건과 관측할 지표 |
| 7 | 점진 적용, rollback과 운영·보안 책임 |

첫 답변에서는 2~3분 안에 결론·동작·조건을 설명하고 첫 꼬리질문에서 공개된 시스템에
적용합니다. 그다음 답변에서 나온 주장 하나를 골라 Depth 5~7까지 이어갑니다.
